# .ai-rules.md

AI Editing Rules

Project: **The Remote Worker’s Survival Guide in the AI Age**
Author: Gary G. Bayes, BABA, MBA

Purpose:
This file defines the **operational rules for AI assistants** working with this repository.

All AI systems must follow these rules when editing, restructuring, or reviewing the manuscript.

These rules exist to:

• prevent hallucinations
• protect manuscript structure
• maintain build compatibility
• preserve the author’s voice

---

# Rule 1 — Do Not Invent Information

AI assistants must not:

• invent statistics
• fabricate research findings
• create fake studies
• create fictional sources

If a claim cannot be verified:

• flag it for review
• recommend a citation

Do not fabricate citations.

---

# Rule 2 — Do Not Invent Tools

When discussing AI tools:

Only reference **widely known, real tools** unless instructed otherwise.

Acceptable examples:

ChatGPT
Notion AI
Canva
Zapier
Perplexity
Otter
Descript

Do not invent tools or companies.

---

# Rule 3 — Preserve the Author’s Message

The book’s core message must remain unchanged:

Human Judgment

* AI Assistance
* Smart Workflows
  = **Leverage**

AI assistants must not:

• shift the theme
• introduce contradictory arguments
• change the audience focus

---

# Rule 4 — Respect the Book Architecture

The manuscript follows this structure:

Front Matter

Part I — Understanding the AI Era
Part II — Applying AI to Daily Work
Part III — The Remote Worker Toolkit

AI assistants must not:

• add new parts
• remove chapters
• merge chapters

unless explicitly instructed.

---

# Rule 5 — Maintain Chapter Structure

Standard chapter structure:

Opening Hook

Concept Explanation

Visual Diagram

Example or Case Study

Key Insight

Chapter Takeaways

Action Plan

Transition

AI editors should maintain this structure whenever possible.

---

# Rule 6 — Visual Placement Rules

When inserting visuals:

• use existing files only
• reference files listed in `.visual-map.md`
• place visuals after concept explanations

Required format:

```id="fwg9lt"
![Visual Title](../../visuals/path/visual-file.svg)
```

Every visual must include:

• caption
• interpretation paragraph

Do not invent visual filenames.

---

# Rule 7 — Markdown Compatibility

The manuscript must remain compatible with:

Pandoc
EPUB export
MkDocs website

AI editors must:

• use Markdown syntax only
• avoid HTML formatting
• avoid emojis or Unicode symbols
• keep headings consistent

Use:

```id="r56hl1"
## Section Title
```

Avoid excessive heading levels.

---

# Rule 8 — Maintain Build Compatibility

The manuscript must build successfully using:

scripts/build-book.sh

AI assistants must not:

• modify build scripts
• introduce unsupported Markdown features
• break relative image paths

If structural changes affect builds, flag them for review.

---

# Rule 9 — Case Study Rules

Examples in the book should be:

• fictional
• realistic
• clearly labeled

Example label:

*Fictional example based on common remote work situations.*

Do not present fictional examples as real companies or individuals.

---

# Rule 10 — Prompt Library Standards

Prompt entries must:

• be copyable
• remain concise
• include clear instructions

Prompts should:

• avoid unnecessary technical complexity
• encourage reviewing AI output

Example format:

Prompt:
Instruction text

---

# Rule 11 — Tone Consistency

The tone should remain:

• calm
• practical
• professional
• accessible

Avoid:

• hype language
• dramatic predictions
• exaggerated productivity claims

Preferred style:

clear explanations
realistic advice
practical examples

---

# Rule 12 — Avoid Redundancy

AI assistants should:

• remove repetitive explanations
• simplify overly long passages
• avoid repeating the same idea across chapters

When redundancy is detected:

suggest consolidation rather than duplication.

---

# Rule 13 — Respect Supporting Files

Before making structural decisions, AI assistants should review:

.ai-context.md
.bookmap.md
.chapter-map.md
.visual-map.md
.workflow-map.md

These files define the architecture of the book.

---

# Rule 14 — Preserve the Author’s Voice

The author’s voice should feel like:

an experienced professional explaining practical workflows.

Avoid:

• academic tone
• overly technical explanations
• marketing-style writing

---

# Rule 15 — When Uncertain

If an AI assistant is unsure about a change:

• flag the issue
• suggest alternatives
• request clarification

Do not guess.

---

# AI Editing Priority

When editing the manuscript, follow this priority order:

1. Accuracy
2. Structural consistency
3. Reader clarity
4. Tone and style
5. Minor wording improvements
